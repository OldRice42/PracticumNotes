  1. Always write just the required lines of code, and try not to touch anything besides that. If you do touch some unrelated code it might cause problems during lesson submission.
  2. If lesson consists of multiple parts, do not remove your code from one of the previous part.

Maksim Kulichenko, May 4th, 2020, 5:07 PM

Hey, Everyone!

Understanding BEM can be difficult, I know that :upside_down_face: To help you with the first project I'll try to summarize it here in simple words:

BEM is a methodology. It allows us to reuse styles and html, and avoid css class collisions (when you have 2+ classes with the same name in your project).

Block - is a basic entity. The classname for this entity should answer the question - "What is it?".

Structure for a class name is - {BLOCK_NAME}

Examples classes are - button, footer, gallery

Element - this entity should always be part of a block. The class name for this entity should answer the question - "What is it?". The same as for the block. The only difference is that elements are considered unusable without it's block. For example, you could have a form block, and it's elements - form__input and form__button. Those elements are only making sense when they are part of a form block. They can't be used without it.

Modifier - this entity can't be independent, it should always be used with a block or an element. The class name should answer the question - "How does it look?" (button_theme_white, button_theme_black) or "In what state is it?"(button_disabled, button_focused).

Structure for a class name is -

{BLOCK_NAME}__{ELEMENT_NAME}_{MODIFIER_KEY}_{MODIFIER_VALUE} or {BLOCK_NAME}__{ELEMENT_NAME}_{MODIFIER_STATE} for elements; Example classes are: button__icon_size_m , button__icon_hidden

{BLOCK_NAME}__{MODIFIER_KEY}_{MODIFIER_VALUE or {BLOCK_NAME}_{MODIFIER_STATE} for blocks; Example classes are: button__size_m , button__hovered

Here is also a helpful link that can help you understand BEM naming convention better: 9elements.com: BEM Naming Cheat Sheet
