Git for Beginners
Intro to Git

You’re starting to get familiar with front-end development. Now it’s time for your first independent project. It'll look even more like the projects you’ll see in the real world. You’ll have to write a lot of HTML and CSS. We'll also get to know, bit by bit, the tools that web developers use daily. We'll focus on Git first. Before we take a look at it, let’s look at what it’s used for.

Version Control

Imagine that you’ve successfully created a website for a client. The website is running, and people are visiting it. The client asks you to redo part of the website’s functionality. They believe that it’ll make using the website easier. You make the changes, update everything, then leave it and forget it. A month later, the client writes to tell you that their assumption was wrong, and they want to revert the website back to its previous state. But you don’t have the previous version saved, so you have to write everything from scratch again.

This wouldn’t happen if there were a way to save file versions from different times. You could just revert to the previous version without needing to redo anything. Imagine that the entire file history has been saved and that you can revert to code from the past:

Guess what? Git knows how to handle tasks like this. We can save different versions of the code as we write and rewrite it. That’s why Git is called a version control system.

Teamwork

You’ve been working on your project and tasks by yourself. In real life, however, you’ll be expected to write code as part of a team. Several people can work on one website.

You could build it with a friend, for instance. Every time you add changes to the code, you could send your friend an updated archive. Then your friend could download it, put in some changes, and send it back to you. But then you decide to put in your changes before your friend sends back his version. After all, it’s quicker to work in parallel.

The two of you end up sending each other archives with changes at the same time. How do you combine them? There are lots of changes, and neither one of you remembers exactly what you changed. You have to open your code on the left, your friend’s code on the right, and compare everything manually. Don't want to work that way? Don’t sweat it. Git will do it automatically.

Summary

The ability to save and merge team members' work makes Git an indispensable tool when working on code as a team. Most employers require you to know how to use it. That’s why we integrate it into projects from the get-go. This chapter is about Git.

Installing Git

You have to install Git, just as you would any other software. What you have to do varies from one operating system to another. Select your OS.

Windows

Download and install Git on your computer. Link to the download page on Git’s official website.

When you launch the installer, you’ll have the option to install Git Bash, too. This is another program that helps Git run correctly. Be sure to check the box to install it. Check “Use Git from the Command Prompt,” too. We’ll discuss this further in the next lesson.

macOS

Download and install Git on your computer. Link to the download page on Git’s official website.

It’s pretty simple, except for one thing. The installer will offer you several ways to use Git. Select “Use Git from the Command Prompt.” We’ll cover that in more detail in the next lesson.

Linux

Download and install Git on your computer. Link to the download page on Git’s official website.

Use the Terminal to install it. Search for the Terminal and select it from the list of programs. The Terminal lets you use only text commands to interact with the program. Search for the Install Git command for your version of Linux, paste it in the Terminal, and press Enter. Congratulations! You’ve installed Git.

Command Line

We’ve gotten used to software having graphical interfaces with buttons and drop-down menus with settings. The command line is also an interface. It just happens to be a text-based one. You have to enter commands to work with programs. For example, to create a folder, you usually right-click and select the option you need in the graphical menu. But you can also create a folder with the command prompt by entering the appropriate command and the folder name. It’s just a different kind of interface.

Now let’s get down to business. To open the command prompt:

  • On Windows, open GIt Bash.
  • On macOS or Linux, open Terminal.

We talked about creating a folder in our example, so let’s explore how to do that in the command prompt.

In the command prompt, you’re always in a folder, too. You just can’t see it. The pwd command lets you find out where you currently are. PWD stands for print working directory. Enter pwd in the command prompt and press Enter:

pwd

/Users/students_yandex

The command prompt returns a response. It’s the path to the folder we’re in right now. You’ll see your username instead of students_yandex. This is the home directory, or the directory where the user stores their personal files. When you open the command prompt, this is where you start off.

In a graphical interface, you see the open folder’s content. Use the ls command to do this in the command prompt. The command is used to list directory contents. If you type this command and press Enter, you’ll see the current directory’s contents: its files and folders.

So now we know where we are. We’ve had a look around, and now it’s time to learn how to move around. To go from one folder to another, use the cd command. This stands for change directory. The syntax for this command is cd file_name:

cd Desktop # going to the desktop

pwd

/Users/students_yandex/Desktop # this is where we are now

If the folder name has spaces, use quotation marks:

cd "Desk top"

To return to the parent directory—that is, one level higher—write two dots .. instead of the folder name:

cd .. # going one level higher

pwd

/Users/students_yandex # we're back

Why two dots? Who knows?!

Creating folders and files

Now let’s do something. The mkdir command creates a folder. Assign the name of the new folder to the command.

Create a folder called dev in the home directory.. This will be where you save your school projects:

mkdir dev # will create the dev folder in the current directory

Go to the dev folder, create a folder called first-project inside it, and go to that folder:

cd dev

mkdir first-project

cd first-project

Let’s create the files we need. We can do this with the touch command, assigning it the file name, for example:

touch index.html # will create the index.html file in the current folder

Create the style.css file on your own.

By the end of this lesson, your home directory should have the dev folder with the first-project folder inside it and two files inside that, index.html and style.css.

Deleting folders and files

The last thing we’ll do is learn how to use the command prompt to delete files and folders. To delete files, use the rm command, which stands for “remove”:

rm about.html # will delete about.html from the current folder

If, however, the folder you’re trying to delete has files in it, the command prompt won’t delete it. Instead, it’ll give you a message that the folder is not empty:

This measure is in place to keep you from accidentally deleting the files you need. If you still need to delete the folder, use the rm command, like this:

rm -r images # will delete the images folder from the current folder

In this case, we call -r a flag. We say, “We called the rm command with the r flag.” This flag is responsible for recursively deleting files and folders. That means that the directory’s entire contents will be deleted. Be careful: deleting files with the rm and rmdir commands is irreversible. The files don’t go to the recycle bin; they’re permanently deleted.

Cheat Sheet Command Line

We’ve prepared a cheat sheet for you. If you forget any commands, remember to come back to this lesson. We didn’t cover all of the commands on the list, so give them a quick run-through even if you remember the previous lesson well.

  • pwd shows you what folder you’re in
  • ls shows files in the folder you’re in
  • cd first-project go to the first-project folder
  • cd first-project/< html > go to the html folder in the first-project folder
  • cd .. go up one level to the parent folder
  • cd ~ go to your home directory (that’s /Users/students_yandex in our case)
  • rm about.html deletes the about.html file
  • rmdir images deletes the images folder
  • rm -r second-project deletes the second-project folder and everything in it
  • touch index.htm style.css script.js if several files need to be created, their names can be separated with spaces

To avoid having to enter file and folder names in full, you can start to enter their name and press the Tab key. The command prompt will finish writing the directory on its own if the corresponding file or folder is in the current directory.

For example, if you’re in the dev folder, start typing cd first and press Tab. If the first-project folder is inside dev, the command prompt will automatically complete its name. You’ll just need to press Enter.

Configuring Git

You’re now a command prompt Jedi. In this lesson, we’re going to use it to configure Git. You need to choose a name for yourself, just like in a computer game. This will help other developers understand what changes you made. Let’s do that and get things underway with Git.

Configuration

If you’re working on macOS or Linux, launch the Terminal. If you’re using Windows, launch Git Bash. Now input your name and email address. To do that, go to the command line and run the command git config with the option --global. Input your info as the values for user.name and user.email:

git config --global user.name "Students Yandex"

# enter your first and last name or alias using Latin letters inside quotation marks

git config --global user.email studentsyandex@gmail.com

# enter your real email address here

Now check to see what you get. Run the command git config with the option --list.

git config --list

# use the command prompt to get the list of all config properties

In response, the command prompt will display your configurations:

user.name=Students Yandex

user.email=studentsyandex@gmail.com

Linux and macOS users will need to exit file view by pressing the Q key.

Where configuration settings are stored

If you need to view or change Git configuration settings, it doesn’t matter which command prompt directory you’re in. Calling git config --global will work from wherever you are.

There it is! The first configuration settings are the most important because you can now fully use Git.

Connecting Git to the Project

We need to turn our project folder into a git repository, so that Git can start working with it. That means that Git will start tracking all changes made inside that directory. To do that, go to the folder containing the project and enter the command:

git init

The command prompt will reply with a message that the repository has been initialized.

Make the first-project folder a git repository. To do that, go to the first-project folder using the cd command, and write git init in the command line.

After that, run the git status command. Git will reply that this folder has two files, index.html and style.css

Git calls these files untracked files. That means that Git sees them, but it won’t be able to save their version right now if you try.

That’s not what we want. Our goal is to learn how to save versions. To do that, add the files first. You’ll need the git add command for this. When using it, indicate the file name that we want to preserve in its current form. For example, git add index.html.

If we want to save the current status of all files, we can jump to the option --all. git add --all, or git add -A for short:

Files highlighted in green are ready to be saved in their current versions. That’s what the git add command tells Git. But we haven’t saved them yet. We have to tell Git what we want to save before we actually save anything.

In Git, saved file versions are called commits. Making a commit means saving the current state of your repository. We’ll tackle this in the next lesson.

If we edit any file right now, it’ll turn red again:

Use git add to make the updated version go to the commit, like this:

git add index.html

Or use the shortened version, git add --all:

git add -all

This will add all modified files to be saved.

Making the First Commit

When all files are ready to be saved, we’ll make our first commit—that is, record the changes we’ve made so far. We do this using the command git commit with the option -m. -m is followed by the name of the commit in quotation marks:

git commit -m "my first commit"

Press Enter and the current version of files will be saved with the name “my first commit.”

“My first commit” isn’t the best name for a commit. Commits should be named so that anyone in the future can tell what changes were made. For example: “HTML and CSS for the home page,” “fixed bug,” “styles updated.”

Make several commits by yourself in the first-project repository. The git log command will show the commit history:

If it doesn’t work, make sure that you called git add and git commit in the right order. If it still doesn’t work, ask a question in Slack. In a real-life situation, though, you’d have to find the answers to a lot of questions on your own. We’ll let that be a small exercise.

Teamwork and GitHub

We finished the last lesson by viewing the history of commits.

That means that we've saved our project several times, and we can jump back to them at any time. We’ll learn how to switch between different versions a little later. For now, let’s discuss Git’s second-biggest feature: making team coding convenient.

Up to now, we’ve used Git locally. To share a repository, we’ll need to get the remote version. There are several platforms that let you do that. The most popular is GitHub, which is what we’ll use.

You have to sign up with GitHub to start using it. This is what the next lesson is about.

GitHub is also a social network for developers. Since its founding in 2008, GitHub, according to its own statistics, has connected tens of millions of people, given them the opportunity to make their dreams a reality, and let them freely exchange information.

With GitHub, you can create an account, publish your code, and work jointly on any open source projects. You can also publish a website for free on GitHub Pages.

Lots of big companies—such as Facebook, Twitter, Google, Apple, and Valve—keep their projects on GitHub. It’s the perfect platform to hone your Git skills.

Signing up

This step is pretty easy. Enter your email address, username, and password. You’ll then be taken to the GitHub interface. Select "Choose Free" to open a free account. You’ll be asked several questions, such as what knowledge and skills you have and what your profession is. You can skip the survey. Just scroll to the end of the web page and click the "Complete Setup" button. You’ll just need to confirm your account. An email with a link to activate your account will have been sent to the email address you indicated. Clicking that link confirms your account.

Summary

You’re signed up now. That wasn’t so bad. You’re now at the screen to create your first remote repository. That’s what we’ll do in the next lesson.

Linking the Local and Remote Repositoris

We signed up on GitHub so we could upload our local repository, first-project to GitHub's remote platform. To do that, we first need to create a repository on GitHub, then link the local repository to it. Let’s start by creating the repository on GitHub.

Creating the repository on GitHub

Go to your profile using the link https://github.com/your-username

. This displays what other users can see about you and your projects. You don’t have any projects yet. You can tell that from the message “You don’t have any public repositories yet”.

Let’s create a repository. Click on Repositories:

Then click "New". GitHub’s interface is in English, so that’s a plus.

The window to create a new repository is now open. Name it first-project. The remote repository’s name doesn’t have to be the same as the folder with the project on your computer. But just to make sure that we don’t get confused, let’s name them the same thing.

After you write the name, click "Create Repository". We don’t need to worry about the other fields right now.

We’ve created the remote repository.

Linking the local and remote repositories

To link the remote repository to the local one, use the git remote add command. Assign two parameters to this command: the name of the remote repository and its URL. Like this (using your URL, not this one):

git remote add origin https://github.com/YandexPracticum/first-project.git

The URL makes sense, but why is the name of the repository origin instead of first-project? Things look a little confusing here. In this case, the repository’s name may not match its name in GitHub. But this is the name we’ll call the remote repository locally when we enter commands in the command prompt. We could assign it the name first-project, but origin is the standard name for a remote repository. That will let us omit it from commands in the future. By default, Git will search for a remote repository with the name origin.

The repositories are linked. Now we just need to upload the code to GitHub. The git push command does this. Call the command like this:

git push -u origin master

The command prompt will ask you for your GitHub username. Input it and press Enter. Then type your GitHub password and press Enter.

Windows users will sometimes need to enter their username and password in the pop-up window (you’ll be able to tell which one), not the command line.

When you enter your password, the characters won’t appear on the screen, but that's just for security. The command line will still get your password once you type it in. If you enter your username and password correctly, the commits will move to GitHub.

We’ll take a more in-depth look at the git push command in the next lesson. In the next sprint, we’ll learn what the word master means. For now, go to the remote repository on GitHub and refresh the web page.

The code has been uploaded (or pushed, as developers say). Now other developers can connect to this repository. You’ll soon find out how to do that. Now study the GitHub interface and the first-project repository on it independently.

Synchronizing the Local and Remote Repositories

You’ve signed up, created a local repository, and pushed it to GitHub. The hard part is behind us. Nice job. Now we have to figure out how to get changes from the remote repository and push any changes we make to it. Let’s talk about that now.

git push

You’ve already gone through the commit cycle. You added a file using the git add command and committed it with a comment using git commit. After that, you linked your local repository to one you made on GitHub using a new command, git push, which you will use from now on to upload, or push, changes you make to your remote repo. This will allow your teammates to see and incorporate your work into their own.

git push -u origin master

We already used it in the previous lesson. We use this command to send new commits to GitHub. Now let’s practice how to do that.

Add some simple markup to the index.html file. Add the files to the commit and make it. If you did everything right, the command prompt window after git status should look like this:

Now it’s time to incorporate the changes into your remote repository. Enter the git push command. The commit that you just made will move to the remote repository. If everything is correct, you’ll see this screen:

Move to the first-project repository in your GitHub account. You’ll see the modified files in the GitHub interface.

git pull

Right now, you’re working on the repository by yourself. But in real projects, you'll be working with others. Imagine you wrote code all day Friday, but decided to wait until Monday to push your commit. Now imagine your coworker worked on your project all weekend long, and by the time you come in on Monday, the repo on Github is very different from the one you have in your commit. What do you do? Use the git pull command to pull the changes your coworker made.

Whenever you’re working on a repository with other people, always run git pull before you publish changes. Somebody else may have published changes while you were writing your part of the code.

Cheat Sheet: Git

You can’t memorize all the commands at once. But the commit cycle is always the same. We’ve put together another cheat sheet in case you forget what comes after what.

Commands for synchronizing your local repository with one on GitHub.

git remote add origin https://github.com/YandexPracticum/first-project.git — when we’re in the folder containing your local repository, this will link it to the remote repository (remember to enter your own URL).

git push -u origin master — push all files from the local repository to the remote one that you linked to it.

Commands to commit the work you want to push.

git add file_name — add a single file by writing its name after add.

git add -A — to commit all files, use -A.

git commit -m "comment on commit" — make the commit. Leave a comment with the saved files to make it easier to understand which changes were made.

Commands to push your commits.

git pull — pull changes made by other developers.

git push — publish changes to a remote repository.

Cloning a Repository

In previous lessons, you created a local repository and then linked it to a remote one. But sometimes, you’ll need to copy a ready-made repository to your computer. This process is referred to as cloning. We’ll cover that now.

git clone

Cloning a repository is usually the first thing a developer does in a new job. Let’s run through a scenario of this situation. Create a second repository in your account and call it second-project.

You’ll see the new repository screen. Copy its address, open the command line, and go to the folder where you’re going to save the repository (that's the dev folder, which you created in a previous lesson). You’re going to enter a command you haven’t seen before, git clone, and the repository address:

git clone https://github.com/students-yandex/second-project.git

# don't copy this repository address; use your own

As soon as cloning has finished, go inside the repository using the cd command, create the files index.html and style.css, and make a commit. Open your GitHub account, select the second-project repository, and you’ll see the published commit:

So now you have two ways of creating a repository at your disposal. But here’s one thing to be aware of: although you can clone absolutely any public repository, you can only push your code to repositories you've created yourself or have permission to push to. If you want to push your work to other repositories, you have to send a pull request.
